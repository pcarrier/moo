Rust host · V8 runtime · SQLite memory · MCP

A calm, local-first harness for curious agents.

Moo bundles a TypeScript agent runtime, a Rust/V8 host, a Solid web shell, and an inspectable SQLite store into one hackable local system. It is built for experimenting with tools, memory, embedded apps, and model adapters without turning the project into a cloud service.

prototype honesty

Moo is powerful because it is not pretending to be safe.

The harness exposes real local capabilities to agents. It does not sandbox processes, enforce approvals, or protect you from a bad tool call. Run it where you are comfortable giving those tools access.

What makes it useful

Small pieces you can understand, replace, and debug.

Moo favors boring storage, explicit APIs, and a visible UI over magic. The goal is a harness that is easy to take apart when an agent does something interesting or weird.

🧰

Orthogonal host APIs

Files, processes, HTTP, env, refs, objects, facts, chats, UI prompts, MCP calls, and events are separate primitives rather than one giant plugin abstraction.

🧠

Memory you can query

Chats and RDF-style facts live in SQLite, so project context can be inspected, joined, summarized, and debugged without a remote datastore.

🪟

A web shell, not a black box

The Solid UI exposes chats, timelines, memory, apps, and MCP setup while the Rust server keeps the local boundary clear.

🧪

Built for experiments

Swap model adapters, add commands, generate embedded apps, or wire a new tool surface without waiting for a platform to approve the shape.

Architecture

One binary, one harness, one local store.

Cargo builds the Rust host, bundles the TypeScript harness, and embeds the web UI. At runtime, agents talk to host ops through a V8 boundary while the browser speaks to the server over HTTP and WebSockets.

  • Rust host owns the filesystem, processes, server, WebSocket, and native ops.
  • TypeScript harness defines the agent-facing moo API and commands.
  • Solid UI renders chats, memory, app panels, and setup screens.
  • SQLite stores durable local state: chats, refs, objects, facts, and memory.

Agent surface

A practical API for local agent work.

Tools are just asynchronous JavaScript. The host capabilities are intentionally direct, which makes them quick to compose and easy to audit.

Embedded apps

Agents can make little interfaces beside the conversation.

Moo explores chat-scoped apps: tiny browser bundles with paired tools and local state. They are rough, but they point at a useful pattern for agent-created workflows that need more than text.

Quick start

Run the local web shell.

With Nix and direnv, the repository shell installs the expected Rust and Bun toolchains. Without them, install Rust and Bun yourself.

terminal
# enter the dev shell
direnv allow

# install UI dependencies
cd web
bun install
cd ..

# build and serve the local web shell
cargo run -- serve

# open http://127.0.0.1:7777

CLI run

cargo run -- run

Eval JSON

cargo run -- eval '{"hello":"moo"}'

Dump state

cargo run -- dump <chat-id>

Hack locally

Use the live dev loop instead of running pieces by hand.

When working on Moo itself, start the repo-level process-compose stack. It keeps the server and web UI moving together so you do not need to juggle separate commands in separate terminals.

Limitations

Know the sharp edges before you run it.

Moo is a prototype for hackers and researchers, not an enterprise agent platform.

No sandbox

Tool calls can reach whatever the underlying process, filesystem, or credentials can reach.

No approval gate

The harness does not add built-in confirmation prompts before a tool executes.

No hosted sync

State is local unless you deliberately wire something else in.

Moving internals

APIs and UI concepts are expected to evolve as the experiment changes.
